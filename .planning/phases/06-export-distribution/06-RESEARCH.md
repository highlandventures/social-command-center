# Phase 6: Export + Distribution - Research

**Researched:** 2026-03-15
**Domain:** PDF generation, email distribution, delivery tracking
**Confidence:** MEDIUM-HIGH

## Summary

Phase 6 adds PDF export and email distribution to the existing report system. The enriched report JSON from Phase 5 (KPIs, charts, executive summary, sentiment themes, recommendations) serves as the data source for both outputs. Chart images are already stored as Vercel Blob URLs from Phase 5, so no new chart rendering is needed -- PDFs and emails reference existing image URLs.

The primary technical risk is `@react-pdf/renderer` server-side rendering in Next.js App Router. The project is App Router only (no `pages/` directory), and `renderToBuffer` has documented compatibility issues with App Router route handlers. However, Next.js 14.2+ with `experimental.serverComponentsExternalPackages` config has confirmed working patterns -- the key constraint is that PDF components must be plain server-compatible components with NO React context or client hooks. The PDF should be generated via a dedicated App Router route handler (`app/api/pdf/[id]/route.js`), not through tRPC (which has request size limits and serialization overhead for binary data).

Email distribution uses `@react-email/components` for template building and `@react-email/render` for HTML compilation, paired with `nodemailer` for SMTP delivery. This is a well-established pattern. The email contains a KPI summary table + executive summary inline, with the PDF as an attachment for full visuals (charts NOT inlined in email per user decision).

**Primary recommendation:** Use a dedicated App Router API route for PDF generation (not tRPC), add `serverComponentsExternalPackages` to next.config.js, pre-fetch chart images as base64 for reliable PDF embedding, and use `@react-email/render` (separate package) alongside `@react-email/components` for email HTML generation.

### Pattern 1: PDF Generation via Dedicated Route Handler
**What:** Use `app/api/pdf/[id]/route.js` as a standalone GET endpoint that generates the PDF, uploads to Vercel Blob, and returns the Blob URL (or redirects to download).
**When to use:** Always for PDF generation -- avoids tRPC binary serialization issues.
**Why not tRPC:** tRPC mutations return JSON. PDF is binary data. A dedicated route handler can stream the buffer directly or return a download URL cleanly.
**Example:**
```javascript
// app/api/pdf/[id]/route.js
import { renderToBuffer } from '@react-pdf/renderer';
import { put } from '@vercel/blob';
import { prisma } from '@/lib/db';
import { ReportPDF } from '@/lib/pdf-renderer';

export async function GET(req, { params }) {
  const report = await prisma.report.findUnique({ where: { id: params.id } });
  if (!report) return new Response('Not found', { status: 404 });

  // Pre-fetch chart images as base64 for reliable embedding
  const content = report.content;
  const chartImages = await prefetchChartImages(content.charts);

  const buffer = await renderToBuffer(
    <ReportPDF report={report} chartImages={chartImages} />
  );

  // Upload to Vercel Blob
  const blob = await put(
    `reports/pdf/${report.id}-${Date.now()}.pdf`,
    buffer,
    { access: 'public', contentType: 'application/pdf' }
  );

  // Log delivery + increment downloads
  await Promise.all([
    prisma.reportDelivery.create({
      data: { reportId: report.id, channel: 'PDF_DOWNLOAD', recipients: [], status: 'SENT' },
    }),
    prisma.report.update({
      where: { id: report.id },
      data: { downloads: { increment: 1 } },
    }),
  ]);

  return Response.json({ url: blob.url });
}
```

### Pattern 3: Pre-fetch Images as Base64
**What:** Before rendering PDF, fetch all chart image URLs and convert to base64 data URIs.
**When to use:** Always -- direct URL loading in @react-pdf/renderer `<Image>` is unreliable with remote URLs due to CORS and network issues.
**Example:**
```javascript
async function prefetchChartImages(charts) {
  const images = {};
  for (const chart of charts || []) {
    if (!chart.imageUrl) continue;
    try {
      const res = await fetch(chart.imageUrl);
      const buf = Buffer.from(await res.arrayBuffer());
      images[chart.id] = `data:image/png;base64,${buf.toString('base64')}`;
    } catch (e) {
      images[chart.id] = null; // Graceful degradation
    }
  }
  return images;
}
```

### Pattern 4: Custom Font Registration
**What:** Register custom fonts at module scope before any PDF rendering.
**When to use:** Required for Figure brand fonts (Sharp Grotesk, Plus Jakarta Sans).
**Example:**
```javascript
import { Font } from '@react-pdf/renderer';

// Register once at module scope
Font.register({
  family: 'SharpGrotesk',
  src: '/brand/fonts/SharpGroteskSmBold21.otf',
});

Font.register({
  family: 'PlusJakartaSans',
  fonts: [
    { src: '/brand/fonts/PlusJakartaSans-SemiBold.ttf', fontWeight: 600 },
    { src: '/brand/fonts/PlusJakartaSans-Bold.ttf', fontWeight: 700 },
  ],
});

// Disable hyphenation for cleaner text
Font.registerHyphenationCallback(word => [word]);
```

**CRITICAL: Font src paths in server-side rendering must be absolute file system paths or URLs, not relative web paths.** In the App Router route handler, use `path.join(process.cwd(), 'public/brand/fonts/SharpGroteskSmBold21.otf')` or a full URL.

### Anti-Patterns to Avoid
- **Using tRPC for PDF download:** tRPC serializes responses as JSON. Binary PDF data should go through a dedicated route handler.
- **Inlining chart images in email HTML:** Email clients handle embedded images inconsistently. Attach the PDF instead (per user decision).
- **Using React context in PDF components:** `@react-pdf/renderer` server-side rendering does not support React context. PDF components must be pure, prop-driven components.
- **Client-side PDF generation with PDFDownloadLink:** While this avoids server-side issues, it means the browser does the heavy lifting, chart image fetching happens client-side (CORS issues), and the PDF cannot be uploaded to Vercel Blob for persistent URLs. Server-side is the right approach per the decisions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PDF page headers/footers | Manual coordinate math per page | @react-pdf/renderer fixed position + render callback | Page numbering ("Page X of Y") requires render callback; library handles pagination |
| Email HTML compatibility | Raw HTML string concatenation | @react-email/components | Email client compatibility (Outlook, Gmail, Apple Mail) requires specific HTML/CSS patterns that the library handles |
| SMTP connection management | Raw net/tls socket handling | nodemailer createTransport | Connection pooling, TLS negotiation, auth handling, retry logic |
| Email address validation | Regex patterns | Zod `z.string().email()` | Already using Zod throughout the project; handles edge cases |
| PDF download triggering | Manual Blob/URL creation | fetch() + URL.createObjectURL on client | Standard browser download pattern |

**Key insight:** Email HTML rendering is deceptively complex -- what works in a browser often breaks in Outlook. @react-email handles this with battle-tested inline styles and table-based layouts.

## Common Pitfalls

### Pitfall 1: @react-pdf/renderer Fails in App Router Without Config
**What goes wrong:** `renderToBuffer` throws `ba.Component is not a constructor` or `PDFDocument is not a constructor` in App Router route handlers.
**Why it happens:** Next.js bundles the package incorrectly for server components.
**How to avoid:** Add to `next.config.js`:
```javascript
const nextConfig = {
  experimental: {
    serverComponentsExternalPackages: ['@react-pdf/renderer'],
  },
};
```
**Warning signs:** Any error mentioning constructors or module resolution during PDF generation.

### Pitfall 2: Font Registration with Relative Paths
**What goes wrong:** Fonts fail to load in server-side rendering because relative paths like `/brand/fonts/...` resolve differently in Node.js vs browser.
**Why it happens:** Server-side code runs in Node.js where file paths need to be absolute filesystem paths, not web-relative paths.
**How to avoid:** Use `path.join(process.cwd(), 'public/brand/fonts/...')` for local file paths, or use the full deployment URL.
**Warning signs:** "Font not found" errors or text rendering with Helvetica fallback when custom fonts expected.

### Pitfall 3: Remote Image Fetch Failures in PDF
**What goes wrong:** Chart images from Vercel Blob URLs fail to load or render as broken in the PDF.
**Why it happens:** Network latency, CORS restrictions, or DNS resolution issues during server-side rendering.
**How to avoid:** Pre-fetch all chart images as base64 data URIs before calling `renderToBuffer`. This makes rendering self-contained and reliable.
**Warning signs:** Empty image placeholders in generated PDFs, especially in production/deployed environments.

### Pitfall 4: Vercel Serverless Function Timeout
**What goes wrong:** PDF generation exceeds the default 10-second Vercel serverless function timeout.
**Why it happens:** Fetching multiple chart images + rendering a multi-page PDF with custom fonts can take 5-15 seconds.
**How to avoid:** Set `maxDuration` on the route handler. Pre-fetch images in parallel. Consider caching generated PDFs.
```javascript
export const maxDuration = 30; // seconds
```
**Warning signs:** 504 Gateway Timeout errors in production.

### Pitfall 5: nodemailer SMTP Connection in Serverless
**What goes wrong:** SMTP connections are slow to establish on each invocation because Vercel serverless functions are stateless.
**Why it happens:** Each function invocation creates a new TCP connection to the SMTP server.
**How to avoid:** Create the transport outside the handler (module scope) so it can be reused across warm invocations. Use `pool: true` option.
**Warning signs:** Slow email sends (2-5 seconds per email) or intermittent timeout errors.

### Pitfall 6: @react-email/render Not Installed Separately
**What goes wrong:** `import { render } from '@react-email/components'` fails -- the render function is in `@react-email/render`.
**Why it happens:** Common confusion between the two packages. `@react-email/components` provides UI components only.
**How to avoid:** Install both: `npm install @react-email/components @react-email/render`.
**Warning signs:** "render is not exported from @react-email/components" error.

### Email Sender
```javascript
// lib/email-sender.js
import nodemailer from 'nodemailer';
import { render } from '@react-email/render';
import { ReportEmail } from './email-templates/report-email';

// Module-scope transport for connection reuse in warm serverless instances
const transport = nodemailer.createTransport({
  host: process.env.SMTP_HOST,
  port: parseInt(process.env.SMTP_PORT || '587'),
  secure: process.env.SMTP_PORT === '465',
  auth: {
    user: process.env.SMTP_USER,
    pass: process.env.SMTP_PASS,
  },
});

export async function sendReportEmail({ report, recipients, pdfBuffer, appUrl }) {
  const html = await render(<ReportEmail report={report} appUrl={appUrl} />);

  const info = await transport.sendMail({
    from: process.env.SMTP_FROM || 'Social Command <noreply@figure.com>',
    to: recipients.join(', '),
    subject: `Report: ${report.title}`,
    html,
    attachments: pdfBuffer ? [{
      filename: `${report.title.replace(/[^a-zA-Z0-9]/g, '-')}.pdf`,
      content: pdfBuffer,
      contentType: 'application/pdf',
    }] : [],
  });

  return { messageId: info.messageId, accepted: info.accepted, rejected: info.rejected };
}
```

## Open Questions

1. **SMTP Provider Configuration**
   - What we know: STATE.md notes "SMTP provider not yet configured (needed before Phase 6 email delivery)"
   - What's unclear: Which SMTP provider will be used (SendGrid, AWS SES, Mailgun, etc.)
   - Recommendation: Implementation should work with any SMTP provider via env vars. Use a test/sandbox SMTP service (like Ethereal.email or Mailtrap) during development. Document required env vars clearly.

2. **Vercel Function Timeout for PDF Generation**
   - What we know: Vercel Hobby plan has 10s timeout, Pro has 60s. PDF with 3 charts + custom fonts could take 5-15s.
   - What's unclear: Which Vercel plan is in use.
   - Recommendation: Set `maxDuration = 30` on the PDF route handler. Pre-fetch images in parallel to minimize wall time.

3. **Font File Loading in Vercel Deployment**
   - What we know: Font.register works with filesystem paths locally. Vercel bundles files differently.
   - What's unclear: Whether `process.cwd() + '/public/brand/fonts/'` resolves correctly on Vercel.
   - Recommendation: Test with both filesystem paths and full URL paths. Fallback: read font files as buffers and pass to Font.register as `src: { data: buffer }`.
