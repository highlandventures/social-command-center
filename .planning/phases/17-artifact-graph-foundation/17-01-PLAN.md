<objective>
Lay the schema and transactional foundation for the Unified Artifact Graph (v3.0 PRD §5): add `Artifact` + `ArtifactRelationship` tables, add `artifactId` FKs to every existing module table that participates in the graph, backfill artifact rows for existing data (idempotent, dry-runnable), and refactor every module-table `create` mutation to go through a single `createWithArtifact` transactional helper so a module row without an artifact row is impossible going forward.

Purpose: Phase 17 is invisible groundwork — no user-visible change. It is the prerequisite for Phases 18–21 (MCP rollout), and for the `hub.*` tRPC router + `/api/artifacts/*` endpoints that ship in 17-02. By the end of 17-01, the graph is populated, the invariant is enforced, but no new query surface is exposed.

Output: `Artifact` + `ArtifactRelationship` Prisma models with enums; `artifactId String? @unique` on 5 module tables; `backfill-artifacts.js` script with `--dry-run`; `createWithArtifact` helper library; 5 module routers refactored to use it; unit tests for the helper and backfill.
</objective>

<decisions>
<!-- Resolutions to PRD §12 Open Questions, grounded in the code above -->

1. **ID normalization.** All existing module tables use `cuid()`, not UUID. `Artifact.id` will use `cuid()` for consistency. FK columns are plain `String`. No ID-format migration needed. (PRD §12 "ID normalization" resolved.)

2. **RLS policy design.** There is no database RLS in this codebase (Neon Postgres, not Supabase). Artifact-level permissions derive from the owning module's existing tRPC authorization — the artifact row inherits its writer's identity via `ownerId` and the graph surface uses the same `protectedProcedure` / `internalProcedure` gates. No `pg_policies` added in 17-01. The 17-02 plan will encode per-module write rules in the `hub.*` router (e.g., `hub.update_artifact` delegates to the owning module's update; AGENCY cannot write). (PRD §12 "RLS policy design" resolved for this codebase. If/when the project migrates to Supabase, RLS policies port cleanly from these app-level rules.)

3. **SSO + TEAM_PASSWORD.** Current auth is Google OAuth via NextAuth, not `TEAM_PASSWORD`. The PRD's "Google Workspace SSO" requirement is already satisfied for the web UI. MCP token issuance (Phase 18) inherits the existing NextAuth session but needs two additions that are OUT OF SCOPE for 17-01: (a) a `@figure.com` domain allowlist in `next-auth-options.js:signIn`, currently allowing all Google accounts, and (b) a new `McpToken` model for per-user bearer tokens. Flagged in the punch list. (PRD §12 "SSO provider" resolved; `TEAM_PASSWORD` is a ghost from a prior credentials-provider era documented in a stale README.)

4. **Transactional invariant.** No existing router uses `prisma.$transaction` for module-row creates. 17-01 introduces `createWithArtifact(tx, { module, type, ownerId, moduleCreate })` that takes the Prisma transactional client and returns `{ moduleRow, artifact }`. Every module `create` mutation is refactored to call it inside `prisma.$transaction`. A module row without an artifact row becomes physically impossible for new writes; backfill closes the gap for historical rows. (PRD §5.5 invariant satisfied.)

5. **Module-to-type mapping** (PRD §5.1 types applied to the actual schema):
   | Module table        | Module string | Default artifact type                                      | Notes                                                                                         |
   | ------------------- | ------------- | ---------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
   | `Post`              | `SOCIAL`      | `POST`                                                     | Owner = `createdById`; parent = `threadParent` (Post.threadId) when set                       |
   | `GtmProject`        | `GTM`         | `PROJECT` (or `CAMPAIGN` when caller sets `isCampaign`)    | Owner = `ownerId`; see decision 7 below                                                       |
   | `GtmTask`           | `GTM`         | `TASK`                                                     | Owner = `ownerId ?? createdBy`; parent = project's artifact                                   |
   | `GtmMoment`         | `GTM`         | `CAMPAIGN` or `MILESTONE` (per `GtmMomentType`, see below) | Parent = project's artifact if `projectId` set, else parent moment's artifact                 |
   | `EmailCampaign`     | `EMAIL`       | `EMAIL`                                                    | Owner = `createdById`                                                                         |
   | `NotionTaskInbox`   | `LC_REVIEW`   | `LC_TICKET`                                                | Owner = `userId`; bidirectional Notion sync already exists (`/api/cron/notion-sync`)          |
   | `Report`            | `SOCIAL`      | `REPORT`                                                   | Owner = `createdById`; generated analytical reports                                           |
   | `HomeTask`          | `HUB`         | `TASK`                                                     | Owner = `userId`; personal task tracker (confirmed in-scope by Miso 2026-04-17)               |
   | `IntelligenceTask`  | `HUB`         | `TASK`                                                     | Owner = `null` (schema has `assignedRole` enum, no user FK); parent = `WeeklyBriefing`        |
   | `WeeklyBriefing`    | `HUB`         | `REPORT`                                                   | Owner = `null` (system-generated); parent of its IntelligenceTasks                            |

   GtmMomentType → ArtifactType sub-map:
   - `LAUNCH`, `TENTPOLE`, `CAMPAIGN` → `CAMPAIGN`
   - `EVENT`, `MILESTONE`, `ACTIVATION` → `MILESTONE`

   Adds `MILESTONE` as a new `ArtifactType` enum value (11 total, up from the PRD's 10). Semantic fit for moments is clean.

   EXCLUDED (not artifacts): `EmailTemplate`, `EmailList`, `GtmProduct`, `GtmOkr`, `GtmKeyResult` — reusable assets / org-level artifacts without a workflow lifecycle. Confirmed with Miso.

   Parent/child edges written at backfill AND runtime:
   - `Post.threadId` → Post (thread structure)
   - `GtmTask.projectId` → GtmProject
   - `GtmMoment.projectId` → GtmProject (preferred when both present)
   - `GtmMoment.parentMomentId` → GtmMoment (when no projectId)
   - `IntelligenceTask.briefingId` → WeeklyBriefing

6. **`@figure.com` domain lock.** `next-auth-options.js:signIn` currently returns `true` for every Google account. Restrict to an allowlist (`@figure.com` plus an explicit `ALLOWED_EMAIL_DOMAINS` env var for approved agency/partner domains). Rejected emails land on the existing `/auth/signin` page with a clear error. In-scope for 17-01 (Task 5) — Miso confirmed this must land before Phase 18.

7. **GtmProject-as-campaign signaling.** Recommendation (accepted pending Miso review): add an optional `isCampaign: boolean` input to `gtmProjects.create` (and a follow-up `hub.update_artifact` mutation in 17-02 that can flip `type` on an existing artifact with an admin check). Default stays `PROJECT`. This keeps the `GtmProject` table clean (no new schema field), makes the override reversible from the UI/MCP, and lets Phase 19's `hub.create_meta_project` composite flag campaigns at create-time.

8. **Future Supabase RLS port.** 17-02 will include a single declarative policy file (`apps/social/lib/artifacts/policies.js`) that encodes per-module read/write rules for the artifact graph. Today it compiles to tRPC middleware checks; the same policy dict translates cleanly to `pg_policies` if/when the project migrates to Supabase. Scope for 17-02, flagged here so 17-01 lays the module/role shape the policy file will key on.

  title      String

  targetId         String

  createdById      String?

  archivedAt       DateTime?

  target    Artifact @relation("ArtifactEdgeTarget", fields: [targetId], references: [id], onDelete: Cascade)
  createdBy User?    @relation("artifactEdgeAuthor", fields: [createdById], references: [id], onDelete: SetNull)

  @@unique([sourceId, targetId, relationshipType])
  @@index([sourceId, relationshipType])
  @@index([targetId, relationshipType])
  @@index([archivedAt])
  @@map("artifact_relationships")
