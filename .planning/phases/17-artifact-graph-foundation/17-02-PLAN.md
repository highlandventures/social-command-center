<objective>
Add the graph-traversal and graph-mutation surface to the artifact schema laid down in 17-01. Ship a `hub.*` tRPC router with 10 procedures (6 reads, 4 writes) and a declarative policy file that centralizes per-module read/write rules. The policy file is written so it compiles to tRPC middleware today but translates mechanically to Postgres `pg_policies` if/when the project migrates to Supabase. No REST endpoints are added ‚Äî the Phase 18 MCP will call the `hub.*` procedures via `createCaller` inside the same Next.js app, which has no HTTP overhead and matches the repo's existing tRPC-only API convention.

After 17-02 ships, the graph is live but invisible ‚Äî no UI consumes it yet. Phase 18's Social MCP is the first consumer; Phase 20 adds web UI Related/Children/Lineage panes.
</objective>

<decisions>
1. **tRPC-only.** No `/api/artifacts/*` REST endpoints. Confirmed 2026-04-17. Deviates from the PRD's literal text; the Phase 18 MCP will use `createCaller` on the `hub.*` router inside the same Next.js app.

2. **Recursive CTEs via `$queryRaw`** for two procedures ‚Äî `hub.getTree` and `hub.traceLineage`. All other procedures use Prisma client methods. The CTE is parameterized using Prisma's `sql` tag (no string interpolation ‚Äî prevents SQL injection even though all inputs are validated cuids).

3. **Depth cap**: `hub.getTree` default depth 10, max 20. Prevents runaway recursion if an edge cycle ever slips into the graph. Implemented as a `WHERE depth < $max` guard inside the CTE.

4. **Fuzzy search**: Postgres `ILIKE '%q%'` on `Artifact.title` for v1. Good enough at thousands-of-rows scale. If p95 latency exceeds ~100ms in production, upgrade to `pg_trgm` GIN index in a later phase (flagged in summary punch list, not blocked on).

5. **Type flip**: `hub.updateArtifact` accepts optional `type` field; when present, the procedure branches to an `adminProcedure`-gated path via `canFlipType(user)`. Other fields (title, status, metadata) follow the module-level write policy. This preserves PRD ¬ß6.D's "confirmation on destructive ops" spirit without requiring a literal `confirm: true` flag ‚Äî role escalation is the confirmation.

6. **Policy matrix** (initial values, tunable by editing the file):
   | Module      | Read                      | Write              |
   | ----------- | ------------------------- | ------------------ |
   | `SOCIAL`    | ADMIN, INTERNAL, AGENCY   | ADMIN, INTERNAL    |
   | `GTM`       | ADMIN, INTERNAL, AGENCY   | ADMIN, INTERNAL    |
   | `EMAIL`     | ADMIN, INTERNAL           | ADMIN, INTERNAL    |
   | `LC_REVIEW` | ADMIN, INTERNAL           | ADMIN, INTERNAL    |
   | `HUB`       | ADMIN, INTERNAL, AGENCY   | ADMIN, INTERNAL    |
   Plus an ownership override: any user can read/write their own artifact (`ownerId === user.id`) regardless of role. AGENCY can still read SOCIAL/GTM/HUB but never write anywhere unless they own the row.

7. **Soft-delete semantics**: `hub.unlink` and `hub.archiveArtifact` set `archivedAt = now()`. They never hard-delete. Queries filter `archivedAt IS NULL` by default, with an optional `includeArchived: true` input flag on reads.
</decisions>

         ```sql
         WITH RECURSIVE tree AS (
           SELECT a.*, 0 AS depth
           FROM artifacts a
           WHERE a.id = ${rootId} AND a.archived_at IS NULL

           UNION ALL

           SELECT a.*, t.depth + 1
           FROM artifacts a
           JOIN tree t ON (a.parent_id = t.id OR a.id IN (
             SELECT target_id FROM artifact_relationships r
             WHERE r.source_id = t.id AND r.relationship_type = 'PARENT_OF' AND r.archived_at IS NULL
           ))
           WHERE t.depth < ${maxDepth} AND a.archived_at IS NULL
         )
         SELECT * FROM tree;
         ```

       - `lineageOf` CTE sketch:
         ```sql
         WITH RECURSIVE lineage AS (
           SELECT a.*, 0 AS depth
           FROM artifacts a
           WHERE a.id = ${artifactId} AND a.archived_at IS NULL

           UNION ALL

           SELECT a.*, l.depth + 1
           FROM artifacts a
           JOIN artifact_relationships r ON r.source_id = a.id AND r.target_id = l.id
           JOIN lineage l ON TRUE
           WHERE r.relationship_type = 'DERIVED_FROM'
             AND r.archived_at IS NULL
             AND l.depth < ${maxDepth}
             AND a.archived_at IS NULL
         )
         SELECT * FROM lineage ORDER BY depth ASC;
         ```
