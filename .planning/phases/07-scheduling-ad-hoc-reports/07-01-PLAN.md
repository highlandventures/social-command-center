**1. Prisma schema (`prisma/schema.prisma`):**
Add Cadence enum (WEEKLY, MONTHLY, QUARTERLY, YEARLY) and ReportSchedule model per research spec:
- `id` String @id @default(cuid())
- `name` String
- `cadence` Cadence
- `reportType` ReportType @default(WEEKLY_PERFORMANCE)
- `enabled` Boolean @default(true)
- `recipients` Json @default("[]") -- JSON array of email strings
- `createdById` String
- `nextRunAt` DateTime
- `lastRunAt` DateTime?
- `lastReportId` String?
- `createdAt` DateTime @default(now())
- `updatedAt` DateTime @updatedAt
- `@@index([enabled, nextRunAt])`
- `@@map("report_schedules")`

Run `npx prisma db push` after adding the model.

**2. Schedule helpers (`lib/scheduling/schedule-helpers.js`):**
Export two pure functions:
- `computeNextRun(cadence, fromDate)` -- returns next Date based on cadence
- `computeDateRange(cadence, referenceDate)` -- returns { dateStart, dateEnd } lookback window
- `cadenceToReportType(cadence)` -- maps WEEKLY->WEEKLY_PERFORMANCE, MONTHLY->MONTHLY_SUMMARY, QUARTERLY/YEARLY->CUSTOM

**3. tRPC schedules router (`lib/routers/schedules.js`):**
Export `schedulesRouter` with procedures:
- `list` query: returns all schedules ordered by createdAt desc, including nextRunAt, lastRunAt, lastReportId
- `create` mutation: validates name (1-100 chars), cadence (enum), recipients (array of email strings via Zod), computes initial nextRunAt from cadence, creates ReportSchedule
- `update` mutation: accepts id + partial fields (name, cadence, recipients, reportType), recomputes nextRunAt if cadence changes
- `toggle` mutation: accepts id, flips enabled boolean
- `delete` mutation: accepts id, deletes schedule

Register `schedulesRouter` in `lib/routers/app.js` under `schedules` namespace.

**4. Cron route (`app/api/cron/run-schedules/route.js`):**
Follow existing cron pattern (verifyCronAuth, force-dynamic, GET handler):
- `export const maxDuration = 300;` (5 min for Pro plan)
- Query: `prisma.reportSchedule.findMany({ where: { enabled: true, nextRunAt: { lte: now } }, take: 3 })` -- cap at 3 to avoid timeout
- For each due schedule:
  - Compute date range via `computeDateRange(schedule.cadence, now)`
  - Compute benchmark period via `getPreviousPeriod(dateStart, dateEnd)`
  - Call `generateEnrichedReport({ reportType, dateStart, dateEnd, benchmarkPeriod })`
  - Create Report record (title: `${schedule.name} - ${now.toLocaleDateString()}`, status: 'READY')
  - If schedule.recipients.length > 0: call `renderReportPDF(report)` then `sendReportEmail({ report, recipients, pdfBuffer, appUrl })`, log ReportDelivery
  - Advance: update schedule with `lastRunAt: now, lastReportId: report.id, nextRunAt: computeNextRun(cadence, now)`
- Process with `Promise.allSettled` for parallelism (max 3 concurrent)
- Return JSON with `{ ok: true, processed, errors }`
- Guard against double execution: skip if `lastRunAt` is within 60 seconds of now

**5. vercel.json:**
Add new cron entry: `{ "path": "/api/cron/run-schedules", "schedule": "*/15 * * * *" }`

**6. Un-skip tests:**
In `__tests__/lib/scheduling/schedule-manager.test.js`, un-skip the `computeNextRun` and `computeDateRange` describe blocks and write real assertions. Keep the tRPC/cron-level tests as `describe.skip` (they need mocked Prisma which is out of scope for this task).
